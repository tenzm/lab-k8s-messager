Весь код на Go должен писаться по принципам clean-architecture
Структура проекта:
1) cmd/service-name/main.go - входная точка, инициализация всех зависимостей
2) internal/domain - основные доменные сущности сервиса и бизнесовая логика связанная с ними
3) internal/repository/<repository_name>/* - слой репозитория - принимает домен, переводит в dto и выполняет
4) internal/external/<external_name>/* - слой внешних зависимотей, походы во внешние / соседние сервисы. Там могут лежать адаптеры, клиенты и тд (тоже принимают на вход и возврщают доменные сущности)
5) internal/usecase/<users>/* - слой юзкейсов, каждый юзкейс - отдельная структура, с методом NewUseCase, структурами Request, Response, и функцией Handle(ctx, Request) (Response, error)
6) internal/controller/<controller_name>/* - presentation слой (реализуют http контракты, конвертируют в usecase/доменные сущности в дергают юзкейсы)
7) internal/config/config.go - содержат конфигурацию проекта (github.com/caarlos0/env/v11 и _ github.com/joho/godotenv/autoload). Имеет функцию Load() (*Config, error)
8) В каждой папочке которая имеет какие-то зависимости создавать файлик contracts.go и выносить все зависимости туда в виде интерфейсов с желательно одним методом type usersGetter interface { Get(ctx, userId ) (User, error) }, названия интерфейсов должны начинаться с lowercase, чтобы область видимости была областью видимости пакета, интерфейсы должны описываться по месту использования а не реализации
9)  В корне проекта должны обязательно быть файлики: Makefile(make generate(генерация proto), make run(запуск go кода)), Dockerfile (сборка проекта), .env.example - список env-файлов которые требуется использовать)

Требования к сервису на golang:
1) Общение между сервисами должно выполняться посредством http, серверная часть на gin
2) bff-сервис должен объединять апи всех микросервисов в один
3) Сервисы должны быть покрыты тестами и протестированы

Техническое задание:
1) Требуется реализовать месседжер, у которого будет несколько возможностей
    1) Возможность регистрации (без пароля), когда клиент заходит у него спрашивается только -- введите имя
    2) Возможность искать других пользователей и отправлять им сообщения
    3) Возможность отправлять картинки/файлы другим пользователям (картинки должны просто сохряняться в файловую систему с названием uuid(file_id) и индексироваться в бд)
    4) Возможность изменять и удалять сообщения (только автором)

2) Для миграций используй goose
3) Для запросов к бд sqlx
4) Для bff нужна будет ручка с long polling для получения свежих сообщений
5) Напиши простенький frontend и подключи его
